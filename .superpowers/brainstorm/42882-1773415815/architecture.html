DocOCR Architecture Approaches

How should we integrate AWS Textract?

Since this is a Kotlin Multiplatform app, we have a few ways to connect the UI to AWS Textract.

A

Option A: Direct to API Gateway + Lambda

The KMP app talks directly to an AWS API Gateway endpoint, which triggers a Lambda function. The Lambda handles the Textract SDK calls and returns the text.

  • Pros: Secure (no AWS credentials in frontend), clean separation of concerns.
  • Cons: Requires setting up AWS infrastructure before the UI is fully functional.
B

Option B: Backend-for-Frontend (Ktor Server)

We build a Kotlin backend (e.g., Ktor) that the frontend talks to. The Ktor server then handles the AWS Textract SDK calls.

  • Pros: Keeps everything in Kotlin. Easy to add more business logic later.
  • Cons: More code to write and maintain; slightly more complex deployment.
C

Option C: Direct SDK Call (Not Recommended for Web)

Using the AWS JS SDK directly from the browser (via JS interop in KMP).

  • Pros: Fastest to prototype.
  • Cons: Security risk (exposing AWS credentials/Cognito to browser), tricky JS interop with Kotlin Wasm.
My Recommendation: Option A
It's the most standard AWS Serverless practice. We can mock the API response in the UI for now, allowing us to build the beautiful Split View interface immediately without waiting for the backend.
